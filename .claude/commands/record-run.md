# Record Run

Ты фиксируешь результат прогона скила deep-log-audit в CHANGELOG.md.

Все тексты — **на русском**. Технические термины — на английском.

---

## Правила анонимизации

**ОБЯЗАТЕЛЬНО:**
- Имена проектов → `Проект A`, `Проект B`, `Проект C` и т.д.
  - Если в CHANGELOG уже есть прогоны — посмотри, какие буквы заняты, и используй следующую
  - Если это тот же проект, что в предыдущем прогоне (пользователь скажет) — используй ту же букву
- Имена файлов логов → `app.log`, `service.log` (generic)
- Пути → обезличенные
- Имена пользователей, серверов, доменов → убрать
- Никаких реальных названий компаний, продуктов, сервисов

---

## Входные данные

Пользователь передаёт через `$ARGUMENTS` описание прогона в свободной форме. Извлеки:

| Поле | Обязательное | Пример |
|------|-------------|--------|
| **Модель** | ✅ | `sonnet`, `opus`, `haiku`, `gpt-4o` |
| **Проект** | ✅ (анонимизируй!) | `мой-сервис` → `Проект B` |
| **N** (чанков) | если известно | `5` |
| **Результат** | ✅ | УСПЕХ / ПРОВАЛ / ЧАСТИЧНО |
| **Команд** | если известно | `45` (tool executions) |
| **Примечания** | если есть | свободный текст |
| **Проблемы** | если есть | список нарушений |

Если обязательное поле не указано — спроси пользователя одним коротким вопросом.

---

## Шаг 1: Сбор контекста

1. Прочитай `CHANGELOG.md`
2. Найди секцию с меткой `← ТЕКУЩАЯ`
3. Определи:
   - Следующий номер прогона (N+1 от последнего в таблице, строки с `—` не считать)
   - Следующий ID проблемы (P{N+1} от последнего в таблице проблем)
   - Какие буквы проектов уже заняты

---

## Шаг 2: Добавь прогон

Добавь строку в таблицу `## Прогоны` текущей ветки:

```
| {N} | {модель} | {Проект X} | {N чанков или —} | {✅ УСПЕХ / ❌ ПРОВАЛ / ⚠️ ЧАСТИЧНО} | {команд или —} | {примечания} |
```

---

## Шаг 3: Добавь проблемы (если есть)

Если пользователь описал проблемы / нарушения — добавь каждую в таблицу `## Проблемы`:

```
| P{N} | {Run#} | {модель} | {описание проблемы, анонимизированное} | 🔴 открыта |
```

Правила:
- Каждая проблема — отдельная строка
- Привязка к номеру прогона (Run#) обязательна
- Модель дублируется из прогона
- Описание — краткое, но информативное
- Если проблема из предыдущего прогона была исправлена — обнови её статус на `🟢 исправлена`

Если проблема уже существует (похожее описание) — не дублируй, а обнови:
- Добавь ссылку на новый Run# через запятую: `Run#1, Run#3`
- Это значит проблема воспроизводится

---

## Шаг 4: Убери заглушки

Если в таблице прогонов или проблем есть строка-заглушка (`| — | — | — | ...`), удали её — теперь есть реальные данные.

---

## Шаг 5: Запиши

Используй edit tool точечно — заменяй только таблицы прогонов и проблем.

---

## Пример

Ввод пользователя:
```
sonnet, проект billing-service, 5 чанков, ПРОВАЛ, 120 команд. 
Проблемы: агент прочитал лог-файл напрямую; субагент вызвал Read дважды на чанке 3
```

Результат в прогонах:
```
| 2 | sonnet | Проект B | 5 | ❌ ПРОВАЛ | 120 | Агент прочитал лог напрямую |
```

Результат в проблемах:
```
| P1 | 2 | sonnet | Основной агент прочитал лог-файл напрямую вместо координации | 🔴 открыта |
| P2 | 2 | sonnet | Субагент вызвал Read дважды (чанк 3) | 🔴 открыта |
```
