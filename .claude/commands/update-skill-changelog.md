# Update Skill Changelog

Ты — ассистент по ведению CHANGELOG.md для репозитория скила. Твоя задача — привести файл в порядок и актуализировать все секции.

Все тексты в CHANGELOG.md — **на русском**. Технические термины, имена веток, команды — на английском.

---

## Правила анонимизации

**Обязательно при каждом обновлении:**
- Имена проектов → `Проект A`, `Проект B`, `Проект C` и т.д.
- Имена файлов логов → `app.log`, `service.log` и т.д. (generic)
- Пути к файлам → обезличенные (`/logs/app.log`)
- Имена пользователей, серверов, доменов → убрать или заменить на generic
- Никаких реальных названий компаний, продуктов, сервисов

## Шаг 2: Аудит и исправление CHANGELOG

Проверь и исправь по каждому пункту:

### 2.1 Структура каждой ветки

Каждая ветка (секция `# {имя-ветки}`) ДОЛЖНА содержать ровно эти блоки в этом порядке:

```markdown
# {имя-ветки} ← ТЕКУЩАЯ    ← только у активной ветки

**Статус:** `тестируется` | `стабильный` | `устарел` | `заброшен`
**Родитель:** [{имя-родителя}](#{якорь}) | нет (первая версия)
**Ветка:** `{git-ветка}`
**Дата:** ГГГГ-ММ-ДД

## Изменения

- Что изменено в этой версии

## Прогоны

| # | Модель | Проект | N | Результат | Команд | Примечания |
|---|--------|--------|---|-----------|--------|------------|

## Проблемы

| ID | Run# | Модель | Описание | Статус |
|----|------|--------|----------|--------|

## Решение по слиянию

- [ ] В процессе тестирования
```

**Проверь:**
- Метка `← ТЕКУЩАЯ` стоит только у одной секции (текущая ветка)
- Статус соответствует реальности
- Родитель указан корректно и ссылка рабочая
- Ветка совпадает с git
- Нет лишних блоков (описание скила, фазы, категории — это всё в `chunks-log-analyzer.md`)

### 2.2 Таблица прогонов

Формат строки:
```
| {N} | {модель} | {Проект X} | {N чанков} | ✅ УСПЕХ / ❌ ПРОВАЛ / ⚠️ ЧАСТИЧНО | {кол-во команд} | {примечания} |
```

- Проекты анонимизированы: `Проект A`, `Проект B`, ...
- Модели указаны: `haiku`, `sonnet`, `opus`, `gpt-4o` и т.д.
- N — количество чанков (если известно)
- Команд — количество tool executions (если известно)

### 2.3 Таблица проблем

Формат строки:
```
| P{N} | {Run#} | {модель} | {описание} | 🔴 открыта / 🟢 исправлена / ⚪ неактуальна |
```

- Каждая проблема привязана к прогону (Run#) и модели
- ID проблемы: P1, P2, P3, ...
- Нумерация проблем сквозная внутри ветки

### 2.4 Анонимизация

Пройдись по всему файлу и замени любые реальные данные по правилам из секции «Правила анонимизации».

## Шаг 3: Обнови оглавление

Перестрой секцию `## Оглавление` с нуля:

```markdown
## Оглавление

- 📍 [v{N}-имя](#v{N}-имя) — тестируется | ГГГГ-ММ-ДД
- [v{N-1}-имя](#v{N-1}-имя) — устарел | ГГГГ-ММ-ДД
```

- Новейшая версия сверху
- 📍 — только у `← ТЕКУЩАЯ`
- Якоря: нижний регистр, пробелы → дефисы

---

## Шаг 4: Обнови Mermaid-диаграмму

Перестрой `## Дерево версий` на основе всех веток:

```mermaid
graph LR
    v1["v1-name<br/>статус"]:::class --> v2["v2-name<br/>📍 тестируется"]:::current

    classDef current fill:#2d6a4f,color:#fff,stroke:#1b4332
    classDef stable fill:#1d3557,color:#fff,stroke:#0d1b2a
    classDef superseded fill:#555,color:#ccc,stroke:#333
    classDef abandoned fill:#7f1d1d,color:#fca5a5,stroke:#450a0a
```

- Идентификаторы узлов: без дефисов (заменить на `_`)
- Стрелки: от родителя к потомку
- Под диаграммой: `main` ← `v{N}` ← ... ← `v1`

| Статус | Класс |
|--------|-------|
| `тестируется` | `current` |
| `стабильный` | `stable` |
| `устарел` | `superseded` |
| `заброшен` | `abandoned` |
